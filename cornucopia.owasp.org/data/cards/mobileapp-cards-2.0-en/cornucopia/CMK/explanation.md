## Scenario: Ruben can use the app, without modifications, to spread malicious code because methods for transfer and storage do not perform proper data sanitization and validation

Consider a scenario where Ruben discovers the app imports contact data from a CSV file, processes it, and stores it. The import function does not sanitize the contact name field before storing it in a local SQLite database. Ruben crafts a CSV with a contact name containing a SQL injection payload. When the contact is displayed in the UI via a `rawQuery` call, the payload executes, modifying the database. The app was not modified. The malicious code was delivered as data.

1. Data that is stored without sanitization can carry payloads that execute when the data is processed later.
2. Import functions that process user-supplied structured data (CSV, JSON, XML) are a common injection entry point.
3. Data exported by the app and shared with other users can carry injected payloads that execute on the recipients' devices.

### Example

Ruben crafts a contact vCard with a name field containing `'; DROP TABLE contacts; --`. He shares the vCard with the target app user. The app imports the vCard, stores the name in the database via a `rawQuery` call with string concatenation. The SQL executes and drops the contacts table. Ruben did not modify the app. The malicious code was in the data. The app delivered it faithfully to the database engine.

## Threat Modeling

### STRIDE

This scenario falls under **Tampering**.

Ruben uses the app's legitimate data-import functionality as a delivery mechanism for malicious payloads, exploiting insufficient input validation and sanitization to cause data corruption or code execution.

### What can go wrong?

- SQL injection from imported data corrupts or exfiltrates the database.
- JavaScript or script content in data fields executes when rendered in a WebView.
- Path traversal in imported file paths leads to unintended file operations.
- Malicious payloads in shared data propagate to every user who imports the same data.

### What are we going to do about it?

- Validate and sanitize all data at the point of import, regardless of whether it was generated by a trusted source.
- Use parameterized queries for all database operations; never concatenate imported data into SQL strings.
- Sanitize all data fields before rendering in UI components; use `textContent` rather than `innerHTML` for text fields.
- Apply schema validation to structured data imports (JSON, XML, CSV) to reject unexpected fields or formats.
